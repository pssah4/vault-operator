## Findings

### H-1 (HIGH): BackupSecretFilter.KNOWN_SECRET_KEYS is incomplete -> 8 token fields leak in auto-backup

- **CWE-200** Exposure of Sensitive Information
- **Location:** `src/core/backup/BackupSecretFilter.ts:16-34`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** Auto-daily backup ZIPs are produced by `AutoBackupRunner.maybeRunAutoBackup` and pass the live `data.json` through `filterSecretsFromDataJson`. The filter checks against a static set of 16 key names. The manual backup path (`BackupTab.stripSensitiveFields`) strips 8 additional token fields that the filter does not know about:

- `webTools.braveApiKey`
- `webTools.tavilyApiKey`
- `githubCopilotAccessToken`
- `githubCopilotToken`
- `kiloToken`
- `cloudflareApiToken`
- `relayToken`
- `mcpServerToken`

A user who has set any of these tokens and enables auto-daily backup gets unredacted credentials inside the ZIP under `.vault-operator/cache/backups/`. The ZIP is local to the vault, so blast radius is limited to anyone with vault-filesystem access (incl. iCloud-sync recipients, vault-clone targets, leaked Mac backups). The drift between the two filters is silent: no compiler check that the two lists agree.

**Remediation.** Extend KNOWN_SECRET_KEYS with the 8 fields above, plus regression tests that pin the new keys. Long-term: refactor both filters to share a single source of truth.

### M-1 (MEDIUM): validateJs forbidden-pattern scan bypassable via Unicode escapes and indirect access

- **CWE-358** Improperly Implemented Security Check for Standard
- **Location:** `bundled-skills/skill-translator/scripts/translate.js:17-27`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** validateJs enforces 9 forbidden patterns (`eval`, `new Function`, `require`, `__proto__`, `process.env`, `child_process`, `fs.*`, `globalThis.X =`). The patterns are word-boundary literals. An LLM-generated translation can sidestep the scanner via Unicode escapes in identifiers (`eval(...)` is valid JS for `eval(...)` but does not match `\beval\s*\(`), indirect access (`globalThis['e'+'val'](...)`), or string-array construction (`['e','v','a','l'].join('')`).

The actual security boundary is the sandbox itself (esbuild-bundled ESM, isolated context). validateJs is defense in depth. The risk is the false sense of safety: a passing validation does not equal safe code.

**Remediation.** Doc-fix: update JSDoc + SKILL.md body to make the heuristic-only nature explicit ("the sandbox is the actual security boundary; this scanner catches obvious foot-guns only"). Stronger fix (AST-based scanning) deferred to IMP-29-08-XX.

### M-2 (MEDIUM): writeTranslation accepts any targetPath / file-target without path-traversal guard

- **CWE-22** Path Traversal
- **Location:** `bundled-skills/skill-translator/scripts/translate.js:115-141`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** writeTranslation only checks `typeof targetPath === 'string'`. Both `args.targetPath` and each `f.target` flow into `${targetPath}/${f.target}` and `${targetPath}/SKILL.md` without an isUnsafePath guard. A malicious skill (or LLM hallucination) could pass `args.targetPath = '../../etc/poison'` to escape the skills folder, or `args.files[0].target = '../OTHER_SKILL/SKILL.md'` to overwrite a sibling.

The SKILL.md body says targetPath should be `.vault-operator/data/skills/<target-name>`, but that is a recommendation. The runtime accepts anything string-shaped.

**Remediation.** Add an inline isUnsafePath helper (since translate.js is a bundled .js, not TS): reject `targetPath` and each `f.target` if they contain `..` segments or absolute-path prefixes. Test pin: `targetPath='../etc'` and `f.target='../foo.js'` both reject.

### L-1 (LOW): InvokeMcpServerTool's activeMcpServers whitelist is bypassed when activeMcpServers is empty

- **CWE-697** Incorrect Comparison
- **Location:** `src/core/tools/agent/InvokeMcpServerTool.ts:107-114`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** The check is `if (activeMcpServers.length > 0 && !activeMcpServers.includes(serverId))`. When the user has not enabled any MCP servers (`activeMcpServers = []`), the first conjunct is false and the whole guard is skipped. The intent (defense in depth: only reach servers the user enabled) is missed in the empty-list case. Subsequent `mcpClient.callTool` fails for connection reasons, not authorisation reasons.

**Remediation.** Change to fail-closed: `if (!activeMcpServers.includes(serverId))` rejects any serverId when the list is empty.

### L-2 (LOW): approvalKey collision when pluginId contains ":"

- **CWE-1023** Incomplete Comparison with Missing Factors
- **Location:** `src/core/tools/agent/pluginApiAdaptive.ts:59-61`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** `approvalKey('foo:bar', 'baz')` collides with `approvalKey('foo', 'bar:baz')`. In practice pluginId is bounded by Obsidian's manifest-id format and method names are JS identifiers, so collisions are theoretical. Defense in depth: a crafted collision could cross-mix approval counts and promotion decisions.

**Remediation.** Assert `isSafePathSegment(pluginId)` and `isSafePathSegment(method)` before composing the key. Reject early if either fails.

### L-3 (LOW): TRANSLATION.json may leak tokens embedded in user-pasted source URLs

- **CWE-200** Exposure of Sensitive Information
- **Location:** `bundled-skills/skill-translator/scripts/translate.js:73-81`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** `manifest.source.repo` is filled from `args.sourceRepo`. If the URL contains embedded credentials (`https://x-access-token:gho_xxx@github.com/...`), they land verbatim in the on-disk TRANSLATION.json. Local-only blast radius but persistent.

**Remediation.** `sanitizeRepoUrl(url)` helper that parses the URL and strips username/password before persisting. Falls back to the input string for non-URL values.

### L-4 (LOW): maybeAutoPromotePluginApi feeds unvalidated pluginId/method into settings keys

- **CWE-20** Improper Input Validation
- **Location:** `src/core/tool-execution/ToolExecutionPipeline.ts:697-719`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** The auto-promotion hook reads `input.plugin_id` and `input.method` directly without `isSafePathSegment`. Whatever the LLM passed ends up as a key in `settings.pluginApi.approvalCounts` and `safeMethodOverrides`. No direct exploit; settings pollution risk (garbage names persist on disk).

**Remediation.** Add isSafePathSegment guard at the start of `maybeAutoPromotePluginApi`; silent skip if either part fails.

---

### L-5 (LOW): validateJs per-line split misses multiline forbidden patterns

- **CWE-1023** Incomplete Comparison
- **Location:** `bundled-skills/skill-translator/scripts/translate.js:36-51`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** validateJs splits on `\n` and tests each line. A multiline pattern like `eval\n  (badcode)` (legal JS) does not match `\beval\s*\(` because the open-paren is on a different line. Same class as M-1 (heuristic incomplete).

**Remediation.** Run the regex against the full source with the `gm` flag and report line numbers via substring search.

---

### L-6 (LOW): translate.js leaves orphan partial writes when a later script fails validation

- **CWE-460** Improper Cleanup on Thrown Exception
- **Location:** `bundled-skills/skill-translator/scripts/translate.js:127-150`
- **Status:** Resolved (fix-loop 2026-05-21)

**Risk.** The write-loop validates and writes each file in order. If script #1 passes + writes, then script #2 fails validation, script #1 sits on disk without SKILL.md or TRANSLATION.json. Inconsistent skill folder state.

**Remediation.** Two-pass: validate ALL scripts first; only enter the write phase if every script passed.

## Positive findings

- isSafePathSegment is consistently used at every InvokeSkillTool / InvokeMcpServerTool input boundary
- CompositionStackService cycle detection includes the type discriminator (skill vs mcp) so identifier collisions across types do not cross-trigger
- buildZip + unpackZip use SHA-256 manifest integrity check; tampered ZIPs are rejected on import
- isUnsafePath helper covers absolute paths, drive-letter prefixes, and `..` segments
- BackupSecretFilter is recursive (catches secret keys nested at any depth)
- 0 npm audit vulnerabilities across 1007 dependencies
- All 4 features have unit tests pinning critical paths (127 new tests in scope, all green)
- `AUTO_BACKUP_SELECTION.exportSecrets = false` is hard-coded so the auto-backup path cannot opt in to secrets via UI
- maxIterations cap on subtasks (FEAT-29-10) prevents runaway sub-skill cost
- AutoBackupRunner persists the timestamp BEFORE pruning so a partial prune does not trigger immediate re-run on the next boot
- InvokeMcpServerTool includes the activeMcpServers whitelist when the list is non-empty (the L-1 finding only applies to the empty-list edge case)
