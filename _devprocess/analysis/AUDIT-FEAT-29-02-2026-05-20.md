## Findings

### M-1 (Medium): Markdown-Tabellen-Korruption durch Plugin-controlled command names

**CWE-1236 Improper Neutralization of Formula Elements (analogous to CSV / Markdown injection)**

**Location:** `src/core/skills/VaultDNAScanner.ts:1062`

**Risk:** `writeCommandsReferenceIfTopPlugin` rendert eine Markdown-Tabelle mit Plugin-controlled `cmd.name`:

```typescript
lines.push(`| \`${cmd.id}\` | ${cmd.name} |`);
```

Wenn ein Plugin einen Command-Namen registriert, der ein Pipe-Zeichen `|` enthaelt (z.B. "Export | with metadata"), wird die Markdown-Tabelle korrumpiert: das Pipe wird als Column-Separator interpretiert und die Tabelle ist falsch gerendert. Newlines wuerden die Tabelle abbrechen.

**Threat-Model:** Kein XSS (Obsidian's Markdown-Renderer macht Whitelist-Sanitisation). Kein RCE. Aber die `references/commands.md` Datei wird vom Agent gelesen ueber `read_file` -- broken table verschlechtert die Agent-Interpretation der Plugin-Commands. Indirekter Quality-Impact.

**Realismus:** Niedrig fuer Top-5-Plugins (deren Command-Namen sind alle pipe-frei). Mittel fuer zukuenftige Top-N-Erweiterungen oder Plugin-Updates, die ungewoehnliche Command-Namen einfuehren.

**Remediation:** Escape pipe und newline in `cmd.name` vor dem Tabellen-Push:

```typescript
function mdEscapeTableCell(s: string): string {
    return s.replace(/\|/g, '\\|').replace(/\n/g, ' ');
}

lines.push(`| \`${cmd.id}\` | ${mdEscapeTableCell(cmd.name)} |`);
```

Gleiches Escape sollte auch in `renderPluginMetadataBlock` fuer die Liste-Items angewandt werden (newlines wuerden die Liste brechen).

**Status:** Resolved (2026-05-20).

### L-1 (Low): Keine pluginId-Validierung in agentFolder-Helpern (Defense-in-Depth)

**CWE-22 Path Traversal (defense-in-depth)**

**Location:** `src/core/utils/agentFolder.ts:124-194`

**Risk:** Die 5 neuen Helper (`getPluginSkillsPath`, `getPluginSkillFolderPath`, `getPluginSkillManifestPath`, `getPluginSkillReadmePath`, `getPluginSkillCommandsRefPath`) joinen `pluginId` direkt in den Pfad. Wenn ein Plugin-Manifest einen pathologischen `id`-Wert wie `../malicious` oder `/etc/passwd` zurueckliefert, wuerde der zusammengebaute Pfad nach `normalizePath` ausserhalb des erwarteten Bereichs liegen.

`vault.adapter` macht typischerweise eigene Pfad-Resolution und limitiert Schreib-Operationen auf den Vault. Aber Defense-in-Depth: validiere `pluginId` am Eingang der Helper.

**Threat-Model:** Plugin-IDs kommen vom Obsidian-Manifest (`manifest.json`). Ein boeses Plugin auf dem User-System koennte einen pathologischen `id`-Wert setzen. Aber wer ein boeses Plugin installiert hat, hat eh schon Code-Execution-Zugriff -- der Trojaner braucht den Helper nicht als Hebel.

**Remediation:** Whitelist-Validator:

```typescript
function assertSafePluginId(id: string): void {
    if (!/^[a-z0-9][a-z0-9_-]*$/i.test(id)) {
        throw new Error(`Invalid plugin id (path-traversal guard): ${id}`);
    }
}
```

Anrufen am Top jedes Plugin-Skill-Helpers.

**Status:** Resolved (2026-05-20).

### L-2 (Low): `renderPluginMetadataBlock` nutzt Plugin-controlled strings ohne Escape

**CWE-1236 Improper Neutralization (analogous)**

**Location:** `src/core/skills/VaultDNAScanner.ts:803`

**Risk:** Die Plugin-Metadata-Section im SKILL.md-Body rendert `${c.id}` und `${c.name}` direkt in Markdown-Liste-Items:

```typescript
lines.push(`- \`${c.id}\` -- ${c.name}`);
```

Wenn `c.name` ein Newline enthaelt, bricht die Liste. Wenn `c.id` einen Backtick enthaelt, bricht der Inline-Code-Block.

**Threat-Model:** Identisch zu M-1 -- Quality-Impact, kein Security-Loch. Plugin-Commands haben normalerweise keine Newlines/Backticks in IDs.

**Remediation:** Gleiches Escape-Pattern wie in M-1:

```typescript
const safeName = c.name.replace(/\n/g, ' ').replace(/`/g, '\\`');
lines.push(`- \`${c.id}\` -- ${safeName}`);
```

**Status:** Resolved (2026-05-20).

## Positive Findings

1. **Layout-Awareness verhindert cross-contamination:** Pre-Welle-1-User behalten 1:1 das alte File-Layout, kein unbeabsichtigter Switch.
2. **`cleanupLegacyPluginSkillsLayout` ist filter-strikt:** nur `.skill.md` und `.readme.md` werden removed. User-Files mit anderen Endungen bleiben erhalten.
3. **`rmdir` ist non-recursive:** der Folder kann nur entfernt werden wenn er leer ist. Erspart die Schaedenklasse von "rm -rf $emptyVar".
4. **`writeFolderFormat` schreibt strikte Anthropic-Frontmatter:** nur `name` und `description`. Removed Felder sind in den Body verschoben, nicht weggeworfen.
5. **Description-Escape:** `description` wird mit `replace(/"/g, '\\"').replace(/\n/g, ' ')` escaped bevor in Frontmatter-YAML eingefuegt.
6. **Idempotenz durch overwrite:** jeder Scan ueberschreibt SKILL.md mit deterministischem Content (Test "is idempotent: a second call produces the identical file content" verifiziert das).
7. **`ensureDirRecursive` schliesst die mkdir-Luecke:** Obsidian's `vault.adapter.mkdir` ist non-recursive; der Helper iteriert pro Segment und schliesst die Luecke fuer das 3-stufige `data/skills/plugin/`-Layout.
8. **Pre-emptive folder-existence-check** verhindert dass `cleanupLegacyPluginSkillsLayout` auf einem fehlenden Folder einen Crash erzeugt.
9. **Try/catch um adapter.remove**: Concurrent-edit-Race wird absorbiert, Cleanup wird beim naechsten Scan nochmal versucht.
10. **Test-Suite deckt beide Pfade ab:** 13 neue Tests in `VaultDNAScanner-writeSkillFile.test.ts` plus 11 neue Helper-Tests, alle gruen.
