# Obsilo Agent v2.4

**Versions 2.4.0 through 2.4.3**, released April 2026.

Introduces automated knowledge maintenance for Obsidian vaults, adds OCR support for scanned PDFs, and improves resilience for large documents and the knowledge database.

## Knowledge maintenance

Knowledge vaults grow fast but rarely stay organized. Notes accumulate without backlinks, frontmatter properties drift from the schema, categories go stale, and attachments keep their camera-roll names. Retrieval quality degrades because isolated notes are invisible to graph-based and semantic search alike.

This release adds a maintenance layer for personal knowledge management: a small number of strong hub notes (topics, concepts) that act as index pages, connected by dense backlinks to the notes they organize. When the agent ingests a new note, it searches for existing hubs first and only creates new entities when none fit. Over time this produces a navigable, well-linked knowledge graph where every note is reachable through its topic cluster and where semantic search benefits from the additional context that backlinks and frontmatter provide.

### Knowledge ingest skill

A bundled skill that guides the agent through a structured integration workflow when you say "integrate this note".

- **Entity recognition:** The agent reads the note and identifies topics, concepts, persons, projects, and organizations mentioned in the text
- **Existing-first search:** For each entity, the agent runs `semantic_search` against the vault. Existing hub notes are always preferred over creating new ones -- the explicit goal is few strong hubs, not many weak ones
- **Proposal and confirmation:** All suggestions (properties, wikilinks, MOC entries, stub notes) are presented to the user before any changes are written. Nothing happens without confirmation
- **Stub notes with content:** When a genuinely new entity is discovered, the agent creates a stub note with a short explanation, key aspects, and links back to the source -- not an empty placeholder but a starting point for further thinking
- **Source handling:** For PDFs and web clips, the agent extracts author, year, and title, proposes a filename following the configured naming convention, and populates frontmatter metadata

### Vault health check

A diagnostic and repair pipeline that runs entirely in code, without LLM calls.

- **Five check types:** orphaned edges in the knowledge graph, missing backlinks between related notes, category mismatches against the ontology, invalid wikilinks pointing to non-existent files, and general graph inconsistencies
- **One-click repair:** A modal shows all findings grouped by type. Click "Repair" and the service fixes them in batch -- backlinks are inserted, orphaned edges are pruned, categories are corrected
- **Checkpoint-backed undo:** Every repair creates a git checkpoint first. If anything goes wrong, click "Undo" to restore the exact previous state. No data loss possible
- **Status badge:** The sidebar shows a health badge with the finding count. Click it to open the repair modal

### Ontology store

A SQLite-backed store for the vault's conceptual schema.

- Stores entity clusters, relationships, and type hierarchies extracted from the knowledge graph
- Supports transitive entity association for improved linking suggestions
- Used by the health check to validate category assignments against the defined schema
- Populated automatically during semantic indexing

### Knowledge maintenance settings

Dedicated settings section for configuring knowledge maintenance features.

- **Category property name:** Which frontmatter property defines the note type (e.g. `Kategorie`, `Category`)
- **Summary property name:** Which property holds the short summary (e.g. `Zusammenfassung`, `Summary`)
- **Source naming convention:** Pattern for source file names (e.g. `Author-Year_Title`)
- **MOC properties:** Configurable properties for Maps of Content
- All settings with i18n support (German and English)

### OCR fallback for scanned PDFs

Scanned PDFs that contain only images (no extractable text) are now handled via Obsidian's text-extractor plugin.

- When `pdfjs-dist` extracts fewer than 50 characters from a PDF, the parser falls back to text-extractor's OCR cache
- The fallback reads from text-extractor's cache directory (`.obsidian/plugins/text-extractor/cache/`) -- no additional API calls
- Previously invisible scanned documents now appear in semantic search results
- Graceful degradation: if text-extractor is not installed or the file is not yet cached, the parser returns the limited pdfjs output with a hint

## Large document ingestion (v2.4.1)

PDFs and Office documents over 50 MB can now be attached and ingested without errors. Previously, a 318-page PDF (64 MB) caused an API error because the full extracted text was embedded in the LLM context window.

- **Context truncation:** Documents larger than 80,000 characters are automatically truncated at page boundaries before being sent to the LLM. A notice tells the agent how to read remaining pages
- **Chunked reading:** `read_document` now supports `start_page` and `end_page` parameters, allowing the agent to read large documents in manageable chunks
- **Attachment support:** Both `read_document` and `ingest_document` accept an `attachment_index` parameter to read from chat attachments, not just vault files
- **Size limits:** Separate limits for images (50 MB) and documents (200 MB). `ingest_document` supports files up to 500 MB since it writes to disk, not to the context window
- **Fallback logic:** When vault file parsing fails (e.g. file too large), `ingest_document` automatically falls back to pre-parsed attachment text

### Input validation hardening

- Numeric string coercion: LLMs sometimes pass tool parameters as strings (`"0"` instead of `0`). The input validator now coerces decimal numeric strings automatically, restricted to simple notation (no scientific notation like `"1e100"`)
- Schema type enforcement: `attachment_index`, `start_page`, and `end_page` parameters use `integer` type instead of `number` to prevent float values
- XML attribute escaping: All file names and paths in attachment XML tags are now escaped to prevent injection via crafted filenames

## Database resilience (v2.4.1)

The knowledge database (sql.js/WASM SQLite) could become corrupted by crashes during writes or iCloud sync conflicts, causing all dependent services (semantic search, graph, ontology, vault health) to fail on next startup.

- **Atomic writes:** Database saves now use a write-tmp-then-rename pattern (on native filesystem) or backup-before-write (on vault adapter), preventing half-written files from corrupting the database
- **Backup on every save:** Each successful save creates a `.bak` copy of the previous version, providing a restore point
- **Integrity check on load:** A test query runs immediately after loading the database. If the B-tree is corrupt, recovery kicks in automatically
- **Auto-recovery chain:** Corrupt primary database? Try the backup. Backup also corrupt? Create a fresh database and rebuild from vault files. No manual intervention needed
- **Stale temp cleanup:** Orphaned `.tmp` files from interrupted writes are cleaned up on startup

## Chat history action buttons

Fixed a bug where assistant messages loaded from chat history had no action buttons. Previously, only active streaming chats showed the "Insert at cursor", "Create note", "Save as Zettel", "Copy response", "Regenerate", and "Delete" buttons. Now these buttons appear on every assistant message, whether from an active chat or loaded from history.



## OpenAI GPT-5 compatibility (v2.4.2)

OpenAI's newer models (GPT-5.4 and later) reject the deprecated `max_tokens` parameter with a 400 error. The plugin now uses `max_completion_tokens` for all direct OpenAI API calls. Other providers (Ollama, LM Studio, OpenRouter) continue to use `max_tokens` for backward compatibility.

## Community plugin compliance (v2.4.2)

Resolved all findings from the Obsidian community plugin review bot:

- **Internationalization:** Migrated all hardcoded German strings in the vault health repair modal to i18n, adding full English and German translations
- **Sentence case enforcement:** Removed all inline ESLint rule suppressions for sentence-case rules (not permitted by the review bot) and rephrased affected strings instead
- **Brand name handling:** Moved provider brand names (OpenAI, OpenRouter) out of the locale module into constants to avoid sentence-case false positives

## Security hardening (v2.4.2)

- **Device auth polling:** Added a maximum attempt limit (10 minutes) to OAuth device-flow polling loops in both Kilo and GitHub Copilot authentication, preventing indefinite background polling if the user abandons the authorization flow
- **Dependency updates:** Upgraded vitest from 2.x to 4.x and vite to 6.4.2, resolving CVE-2026-39365 (path traversal in vite dev server) and 3 additional moderate vulnerabilities in the development toolchain

## Graph intelligence (v2.4.3)

The knowledge graph gains structural analysis capabilities that improve both retrieval quality and vault maintenance.

### Confidence scoring

Graph edges now carry a confidence score. Wikilinks get the highest score (you placed them on purpose), MOC property links are medium, and implicit connections from semantic similarity are lowest. Search results use these scores when ranking graph-expanded results.

### Community detection

Louvain community detection runs at startup to find natural topic clusters in the knowledge graph. The vault health check uses these clusters to spot notes whose category tag doesn't match the cluster they belong to. Uses the graphology library.

### God-node analysis

Vault health checks now flag "god nodes": notes with so many connections that they become bottlenecks rather than useful hubs. Splitting them into more focused notes improves both navigation and search quality.

### Knowledge freshness

Search results now factor in file modification time. A note you edited yesterday ranks slightly higher than one untouched for six months, all else being equal. Older notes still appear, they just don't get the recency boost.

### Batch ingest skill

A bundled skill for integrating a whole folder of notes at once. The agent groups files by topic, shows proposals per group, then applies entity recognition, backlink insertion, and frontmatter enrichment in batch. Same workflow as single-note ingest, just for dozens of files at a time.

## BRAT installation self-provisioning (v2.4.3)

BRAT (Beta Reviewers Auto-update Tool) only installs three files: `main.js`, `manifest.json`, and `styles.css`. All other runtime assets -- worker scripts, bundled skills, PPTX templates -- were missing, causing the knowledge database to fail, the code sandbox to crash, and bundled skills to be unavailable.

- **Build-time embedding:** All runtime assets (workers, skills, templates) are now embedded in `main.js` at build time. Total overhead: 87 KB on a 14.7 MB file (+0.6%)
- **Runtime extraction:** `AssetProvisioner` extracts missing assets to the plugin directory on first load. Version-gated: assets are re-extracted when the plugin version changes (e.g. BRAT update)
- **CDN fallback for WASM:** If `sql-wasm.wasm` is missing on disk, it is downloaded from jsdelivr CDN with SHA-256 integrity verification and cached locally
- **Graceful degradation:** If the knowledge database fails to open, downstream services (semantic search, graph, ontology) are cleanly disabled instead of throwing cascading errors

### Security hardening (AUDIT-010)

- Path traversal guard in AssetProvisioner (rejects `../` in asset paths)
- SHA-256 integrity check for CDN WASM downloads
- Dotfile filter in build-time asset scanning (excludes `.DS_Store` etc.)

## Claude remote integration improvements (v2.4.3)

### Dynamic tool discovery

`execute_vault_op` previously listed 7 hardcoded operations. It now generates the list from the tool registry at runtime, exposing all 33 vault-facing tools to external MCP clients (`vault_health_check`, `semantic_search`, `create_pptx`, and others).

### Vault operations in get_context

The `get_context` response now lists all available vault operations. Since Claude fetches this at the start of every conversation, it always knows what tools exist, even when tool descriptions are cached from a previous version.

## Vault health improvements (v2.4.3)

- **Redesigned repair modal:** Shows detailed findings grouped by type with selective repair. Each finding displays the affected note path, severity, and a description of what will be fixed
- **On-demand scanning:** Vault health check added to the sidebar ellipsis menu for quick access
- **Badge alignment:** Health badge count now shows only repairable findings, matching the modal logic
- **Followup append button:** "+" button on followup suggestions lets users compose replies by appending multiple suggestions

## Other improvements

- **New chat icon:** Changed from generic plus to `message-square-plus` for better visual distinction
- **Documentation:** Added comprehensive settings reference, tools reference, troubleshooting guide, and tutorials for first conversation, getting started, and knowledge workflows
- **Cleanup:** Removed unused i18n locale switching UI, Visual Intelligence settings tab, and RenderPresentationTool (replaced by plan_presentation + create_pptx pipeline)
